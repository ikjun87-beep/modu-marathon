# 모두의 마라톤 — PRD (제품 기획)

> 전신: **달려라 대토신**(친목 러닝 크루 Firebase 웹앱) → **모두의 마라톤**으로 리브랜딩·확장.
> 작성 2026-06-26 · 확장 2026-07-03(레퍼런스 리서치 반영: 캐치테이블/오늘의집/무신사/토스 · Strava/NRC/런데이/Garmin/adidas · HealthKit/Health Connect/Garmin).

## 1. 한 줄 정의
러닝·마라톤을 함께하는 사람들을 위한 **크루 + 기록 앱** — 웹·모바일·스마트워치가 한 데이터로 연동.

## 1-2. 비전 (North Star)
**애플워치·갤럭시워치 + 아이폰·안드로이드의 운동 데이터를 한데 섞는 러닝/마라톤 종합 앱** (나이키 런클럽 급).
- **최우선 목표: 실제 설치·사용(배포/확산).** 사람들이 폰(App Store·Play)에 깔아서 쓰는 것이 성공 기준.
- 데이터 통합: HealthKit(애플) + Health Connect(갤럭시·안드로이드)에서 거리·심박·페이스·경로를 읽어 **한 피드/기록으로 통합**.
- 수익화(후행): **러닝화·용품 판매(커머스) + 광고 + 대회/유료 프로그램 중개**. MVP·사용자 확보 이후.
- 순서 원칙: ① 설치 가능한 실제 앱 → ② 워치 데이터 통합 → ③ 사용자 확보·리텐션 → ④ 커머스·수익화.

## 1-3. 전략적 쐐기 (왜 이길 수 있나) — 리서치 결론
경쟁 앱을 조사한 결론: **"한국 오프라인 러닝 크루 운영"이라는 일(job)을 제대로 가진 앱이 없다.**
- Strava·Garmin = 개인 기록 최강, 크루는 얕음(느슨한 클럽). NRC = 코칭·공유 최강, **크루 객체 없음**(오프라인 이벤트는 앱 밖).
  삼성헬스 = 걸음수 챌린지(러닝 정체성 약함). **런데이 = 초보 솔로 첫 30분**(코칭·칭찬 스탬프)이 강점이나 크루는 온라인 커뮤니티 수준, 디자인 혹평.
- 현실의 크루는 **인스타(모집·브랜딩) + 카톡 오픈채팅(일정 조율) + 네이버 카페(기록 보관)** 3곳으로 흩어져 운영된다 → **이 파편화가 우리가 채울 seam.**
- 우리는 이미 **참석 체크·갤러리·방명록·합류**를 갖고 있다. 여기를 **"특정 크루의 일정·참석·추억·명단의 단 하나의 집"**으로 심화하고, 기록을 연결 조직으로 붙인다.
- 러너가 실제로 원하는 것(근거): **소속감 > 숫자**(adidas 자체 데이터: 48%가 "사회적 연결"이 러닝 주이유), **실시간 응원·안전**, **추억(사진·공유카드)**, **기본은 단순한 기록**.

## 1-4. 포지셔닝 (경쟁 대비 한 문장)
- vs **Strava**: 개인 기록 경쟁이 아니라 **우리 크루의 이번 주 모임**. 세그먼트·유료화 클론 금지.
- vs **NRC/런데이**: 무료 코칭 콘텐츠 군비경쟁 안 함 → **소속·지역·오프라인 운영**으로 차별. (런데이는 초보 코칭, 우리는 크루 — 인접·상보)
- vs **Garmin**: 데이터 밀도 기본값·백분위 무지개 색 지양 → **따뜻하고 단순, 깊이는 점진적 공개(power user만).**
- vs **삼성헬스**: 걸음수가 아니라 **거리·페이스 네이티브 크루 챌린지.**

## 2. 5대 목표 (사용자 지정 · 이번 로드맵의 축)
1. **안드로이드 앱 만들기** — EAS preview(APK)로 실기기 설치부터. iOS는 Apple Developer 후.
2. **홈페이지 완성 + 디자인 강화** — 캐치테이블식 "설득→앱 유도" 제품 홈페이지로 리디자인(§5).
3. **앱 ↔ 홈페이지 연동** — 같은 Firebase 데이터 + 딥링크 + 통일 디자인 + 공유 로그인(§6).
4. **앱 ↔ 워치** — Apple Watch·Galaxy Watch·Garmin·폰 GPS 기록 통합(§7).
5. **향후 서비스 확장** — 앱+홈피가 연결된 라이프스타일 서비스(캐치테이블·오늘의집 모델)로 성장(§8).

## 3. 플랫폼 / 기술 스택 (확정)
- **백엔드: Firebase**(Auth · Firestore · Storage) — 웹·앱·워치 데이터의 **단일 소스**. 스키마 단일 소스 = `app/src/lib/firebase.ts`의 `COLLECTIONS`.
- **웹(홈페이지)**: `web/index.html` 정적 단일 파일 + Firebase JS SDK(CDN). 빌드 없음. (확장 시 Next.js 검토, 현재는 보류)
- **모바일 앱: Expo (React Native) SDK 57 · TS · Expo Router** — `app/`. 한 코드베이스로 iOS+Android(+web 미리보기). Firebase npm.
- **스마트워치(3단계)**: 온디바이스 읽기 — **HealthKit**(애플) + **Health Connect**(갤럭시·안드로이드). Garmin은 클라우드 OAuth(후행). → **EAS dev build 필수**.

## 4. 핵심 기능 (크루 우선 · 리서치 반영)
**크루·소셜 (기존 자산 심화 — 우리의 쐐기)**
- **크루 객체 + 명단 + 역할**(owner/admin/member) — Strava Clubs 모델. NRC·삼성이 없는 지속 컨테이너.
- **모임(그룹 이벤트) + 참석** — Strava Group Event 필드 차용(날짜/시간/**집합장소**/페이스 그룹/정원/코스·사진/타입 Social·Workout·Race). **반복 주간 모임 + 멤버별 참석 이력**(Strava가 못 하는 것 = 우리 edge). 이미 있는 참석을 여기로 심화.
- **크루 피드** = 갤러리 + 방명록 통합(사진·공지·후기). Strava Club Posts 모델.
- **공유 카드**(NRC/adidas 차용) — 크루 단체사진 위에 **거대한 거리·페이스 숫자** + Brand Orange, 원탭 인스타/카톡. **가장 싼 성장 루프**(한국 크루가 이미 수동으로 함).

**러닝·마라톤 (기록 = 연결 조직)**
- **단순 기록 기본**: 거리·시간·페이스·지도·구간(splits) + 사진 1장. 기본은 단순(런데이), 깊이는 점진 공개.
- 크루 랭킹·누적 거리(거리/페이스 네이티브, 주간 리셋) · 마라톤/모임 일정.
- **출석 스트릭(개근) + 효과 기반 칭찬**(런데이 칭찬을 "속도"가 아니라 "나옴"에 적용 — 느린 러너 배려, "every pace has a place").

**계정**
- Firebase Auth(로그인) — 웹·앱 공통 계정. 카톡 오픈채팅 친화 온보딩/초대.

## 5. 목표②③ — 홈페이지 전략 (캐치테이블 모델)
**원칙: 웹 = 검색 유입 + 설득 + 앱 유도 깔때기. 실제 행위는 앱. 단 앱 전용 가치(기록·워치)는 반드시 남긴다.**
- **히어로**: 실제 크루 러닝 장면(사진 확보 전엔 앱 폰목업) + **한 줄 가치 제안** + 1차 CTA(앱/출시알림) + 2차 CTA(카톡 오픈채팅 — 마찰 낮은 즉시 전환).
- **신뢰 숫자 밴드**: 누적 거리 · 크루원 · 함께한 모임 (Firebase 실데이터 자동 집계). 캐치테이블 350만/1만/150만 장치.
- **UGC 갤러리를 핵심 섹션으로**: 크루 러닝 실사진·후기가 곧 SEO 자산 + 광고 소재(오늘의집 '집들이').
- **앱 미리보기 폰 목업**(토스): 크루·러닝 탭 화면을 프레임으로 → "설치하면 이런 걸 쓴다".
- **"이렇게 진행돼요" 3스텝**: 합류 → 모임 참석 → 함께 완주.
- **스티키 다운로드 동선**: 모바일 하단 고정 바 / PC 우상단 + QR. 앱 설치 전용 랜딩 URL 1개(야놀자 `/app/install`).
- **웹·앱 픽셀 통일**(무신사): `DESIGN.md`·`brand.ts` 단일 소스를 웹에 그대로.
- 안티패턴: 스톡 이미지·일러스트 히어로·기능 나열부터 시작 금지. 감정→신뢰→행동 순서.

## 6. 목표③ — 앱 ↔ 홈페이지 연동
- **데이터**: 같은 Firebase 프로젝트/컬렉션(`guestbook·gallery·attendance·runs·waitlist`) — 이미 공용. 웹에서 남긴 참석·방명록이 앱에서 그대로.
- **딥링크/유니버설 링크**: 웹 모임·갤러리 상세 URL → 앱 동일 화면. 미설치 시 deferred deeplink(설치 후 그 화면). Firebase Dynamic Links 만료 → **Expo Router + `expo-linking` universal/app links로 직접 구성.**
- **공유 로그인**: 웹·앱 같은 Firebase Auth(온보딩 게이트 → 로그인으로 확장). 현재는 이름 기반 신원.
- **디자인 연속성**: 색·타이포·카드·컴포넌트 규약 동일 → "같은 서비스" 인식.

## 7. 목표④ — 앱 ↔ 워치 (기술 아키텍처)
**아키텍처: "온디바이스 읽기 → 정규화 → Firestore write"** (소스별 어댑터 → 단일 `Run` 스키마 → `source+sourceId` 멱등 upsert).
```
[Apple Watch]→iPhone HealthKit ─┐
[Galaxy Watch]→Samsung Health→Health Connect ─┤→ app/src/lib/health/ (healthkit.ts·healthconnect.ts·manual.ts→normalize.ts)
[폰 GPS]→expo-location ──────────┘        → Firestore `runs` (dedupe: source+sourceId)
[Garmin]→Garmin Cloud→Webhook→Firebase Function ┘ (Phase 3)
```
- **iOS**: `@kingstinct/react-native-healthkit`(config plugin·TS). 워크아웃/거리/HR/`HKWorkoutRoute`/케이던스. **HealthKit capability + Info.plist 문자열**. 페이스는 파생.
- **Android**: `react-native-health-connect`. `ExerciseSession`·거리·속도·HR·`ExerciseRoute`. **Android 14+는 OS 내장**. ⚠️ route 빈값 이슈 → 실기기 검증 필수.
- **Garmin**: 온디바이스 불가 = **클라우드 OAuth·파트너십 심사·셋업 비용(수천 달러 언급)·백엔드(Cloud Functions) 필요** → **사용자 확보 후로 보류.** 일부는 Health 동기화로 부분 흡수.
- **공통**: HealthKit/Health Connect 도입 순간 **영구 dev build 체제**(Expo Go 불가). GPS 경로는 문서 비대화 방지 → **폴리라인 인코딩 또는 Storage 참조.**
- **통합 `Run` 스키마**(기존 `{name,distanceKm,durationMin,createdAt}` 상위호환): `source`·`sourceId`·`startedAt/endedAt`·`distanceM`·`avgPaceSecPerKm`·`avgHr/maxHr`·`avgCadenceSpm`·`elevationGainM`·`route{encoding,data|storagePath}`·`deviceModel`. (상세: `app/src/lib/firebase.ts` 확장 예정)

## 8. 목표⑤ — 향후 서비스 확장 (앱+홈피 연결 라이프스타일 모델)
캐치테이블(예약)·오늘의집(콘텐츠 커머스) 모델을 러닝에 대입 — **콘텐츠·커뮤니티로 신뢰 먼저, 수익화는 그 위에.**
- **크루 발견·매칭(지역성)**: "○○ 근처 러닝 크루"(당근 "당신 근처") — 지역 SEO 랜딩 + 앱 내 크루 탐색·가입. 오프라인 크루의 모집 채널화.
- **모임/대회 예약·정원 관리**(캐치테이블 예약 대입): 인기 모임 정원·웨이팅·리마인드. 나아가 **마라톤 대회 정보·신청 중개.**
- **실시간 응원·안전 공유**(adidas Live Cheer·Garmin LiveTrack): "지금 뛰는 중" 링크를 크루에 공유, 응원 전송. 안전 겸용(새벽·여성 러너).
- **커머스(러닝화·용품)**: UGC 후기 기반(오늘의집 리뷰 커머스). 기록·발 데이터 기반 추천. 제휴/광고.
- **유료 프로그램·코칭 중개**: 페이스메이커·PT·훈련 프로그램 마켓(우리가 콘텐츠를 만들기보다 중개).
- **UGC 인센티브**: 후기·사진 남기면 배지/스탬프(오늘의집 적립 리뷰) → 초기 콘텐츠 확보 + 광고 소재.
- 원칙: 웹은 발견·설득(SEO 자산), 앱은 실행(예약·기록·결제). 앱 전용 가치를 지운 채 웹에 다 넣지 않는다.
