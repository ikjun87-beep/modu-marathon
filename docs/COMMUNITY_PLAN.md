# COMMUNITY_PLAN.md — 크루 커뮤니티(모임·채팅·투표·챌린지) 기획서

> 작성 2026-07-17 · 회장 요청("당근 모임처럼 채팅·챌린지·채팅 내 투표·모임 생성·운영, 웹에서도 참여").
> 레퍼런스 스크린샷 6장: 당근 채팅목록 / 모임 채팅방 / 투표 / 챌린지 선택 / 모임 챌린지 / 일정+후기.
> **이 문서는 착수 전 의사결정용이다. 회장이 §5 결정 3건을 정해야 코드가 시작된다.**

## 1) 지금 우리가 서 있는 자리 (착수 전 사실관계)

| | 현재 상태 | 커뮤니티에 필요한 것 |
|---|---|---|
| **모임(크루)** | **하나뿐**, 코드에 암묵적("우리 크루") | 여러 모임 or 단일 모임 유지 → **결정 A** |
| **일정** | **하드코딩** (`app/src/lib/events.ts` + `web/index.html` 각각 3건) — 일정 하나 추가하려면 **회장이 코드를 고쳐야** 함 | 앱·웹에서 **일정 만들기** |
| **참석** | `attendance` 컬렉션, `eventId + name` | 그대로 쓰되 uid 추가 |
| **글** | `guestbook`(방명록)·`gallery`(사진)·`comments`(러닝 댓글) | 채팅은 **별도 컬렉션** |
| **신원** | **이름 문자열**. 사칭 가능, delete 전면 개방 | **uid 소유 기반** → 결정 B의 전제 |
| **웹** | 정적 HTML + 같은 Firebase. 참석·방명록·갤러리 가능 | 같은 SDK라 **채팅도 그대로 됨** |
| **Auth** | **2026-07-17 활성화 완료**(익명·이메일). 웹은 `withUid`로 4개 쓰기에 uid를 이미 곁들임(절반 완료) | 앱에도 uid 심기 |

**좋은 소식**: 웹이 정적 HTML이어도 Firebase JS SDK를 그대로 쓰므로, **"폰 접속 어려운 사람도 웹에서 대화·참여"가 공짜로 따라온다.** 회장 구상과 우리 구조가 잘 맞는다. 서버를 새로 만들 필요 없다.

## 2) ⚠️ 먼저 고쳐야 할 것 — 신원 (이걸 안 하면 채팅은 못 연다)

### 2-1. 지금 채팅을 열면 벌어지는 일
문서에 작성자가 **`name` 문자열로 박제**될 뿐 검증이 없다. 그래서 지금 채팅방을 만들면:
- 아무나 **"쫀쫀샷"이라고 사칭해서** 말할 수 있다.
- 아무나 **남의 메시지를 지울 수 있다**(rules `delete: if true` — Auth가 없어 소유 검증 불가라 열어둔 것).
- 방명록은 글 몇 개라 넘어갔지만, **채팅방은 이걸로 무너진다.**

→ **결론: uid 소유 기반이 채팅의 전제다. 순서를 바꿀 수 없다.**

### 2-2. 그런데 uid로 바꾸면 드러나는 함정 3개 (조사 완료)

**함정 ①  게스트 가입 시 uid가 갈린다 — 지금 코드의 실제 결함**
`auth.ts:106`이 `createUserWithEmailAndPassword`를 쓴다 → **새 uid 발급**. 익명 계정과 연결되지 않는다.
계정 카드는 "이메일로 가입하면 기기를 바꿔도 이어집니다"라고 **약속하는데, uid 기반으로 가는 순간 거짓말이 된다**
(게스트로 쓴 글·러닝·참석의 소유권을 가입하면서 통째로 잃음).
→ **수정: `linkWithCredential(EmailAuthProvider.credential(...))`로 익명 계정을 승격**하면 **uid가 유지**된다.
   지금은 신원이 이름 기반이라 증상이 없다. **uid로 넘어가기 전에 반드시 고칠 것.**

**함정 ②  익명 uid는 기기·설치 종속**
앱을 지웠다 깔면 익명 uid가 바뀐다 → 내 글의 소유권 상실(내가 쓴 글을 내가 못 지움).
→ 게스트는 "이 기기에서만 내 것"이 정직한 수준. **이메일 가입을 권해야 하는 진짜 이유**가 여기서 생긴다
   (지금 카피 "기록 지키기"가 비로소 사실이 된다).

**함정 ③  기존 문서엔 uid가 없다**
지금 Firestore의 방명록·참석·러닝·갤러리엔 uid가 없다. 소유 검증을 걸면 **아무도 못 지우는 유령 문서**가 된다.
→ **유예 조항**: `!('uid' in resource.data) || resource.data.uid == request.auth.uid`
   (uid 없는 옛 문서는 종전대로, **새 문서부터 소유 검증**). 구멍은 남지만 새 데이터엔 안 생기고, 옛 문서는 자연 소멸.

### 2-3. 기반 공사 범위 (P0)
- 앱: 모든 쓰기에 `uid` 심기(`crew.add/put`, `saveRun`) — 웹의 `withUid`와 같은 방식
- 앱: **게스트 → 이메일 가입을 `linkWithCredential`으로 교체**(함정 ①)
- rules: `update`·`delete`를 **소유 기반**으로(유예 조항 포함)
- **덤으로 해결됨**: 개명 전파의 동명이인 문제(지금은 남의 동명이인 글까지 같이 바뀜), delete 전면 개방, App Check 부담 경감

> 크기: **작다**(며칠 아님, 반나절~하루 급). 채팅·모임보다 훨씬 작고, **어느 방향으로 가든 반드시 필요**하다.

## 3) 기능별 설계

### 3-1. 일정 만들기 (가장 먼저 할 것 · 가장 싸고 효과 큼)
지금 회장이 **일정 하나 추가하려고 코드를 고쳐야 한다**(`events.ts` + `web/index.html` 두 곳). 이건 명백한 병목이다.

```
events/{eventId}   { crewId?, title, desc, startAt, place, capacity?,
                     uid, name, createdAt }        ← 만든 사람
attendance/{id}    { eventId, name, uid, createdAt }   (기존 유지 + uid)
```
- 하드코딩 `EVENTS`는 **Firestore로 이관**하되, **문서 id를 기존 `ev-0705` 그대로** 두면 기존 참석 기록이 그대로 붙는다(무손실 이관).
- 웹·앱 공용. 웹에서도 일정 생성 가능 → "폰 어려운 사람" 요구 충족.
- **후기**(당근 스크린샷의 "일정 후기")는 `comments`에 `parentId=eventId`로 **이미 되는 구조**. 컬렉션 추가 불필요.

### 3-2. 채팅
```
messages/{id}   { crewId, text, uid, name, createdAt, kind?: 'text'|'poll'|'system' }
```
- Firestore 실시간 리스너로 충분(크루 수십~수백 명 규모). 서버 불필요.
- **비용 주의**: 채팅은 읽기 횟수가 폭증하는 유일한 기능이다. 무료 한도(일 5만 read)는 친목 규모에선 여유롭지만,
  **`limit(50) + orderBy(createdAt desc)`로 최근 것만 구독**할 것. 전체 구독 금지.
- 웹도 같은 리스너로 동일 동작.

### 3-3. 투표 (채팅 내)
```
polls/{id}        { crewId, messageId, q, options: [string], multi?, closesAt, uid, name, createdAt }
polls/{id}/votes/{uid}   { choice: number[] }    ← 문서 id = uid → 1인 1표가 규칙으로 강제됨
```
- **투표는 uid가 없으면 성립 자체가 안 된다**(중복 투표를 막을 방법이 없음). §2의 전제가 가장 강하게 걸리는 기능.
- 집계는 클라이언트에서 `votes` 서브컬렉션 세기(친목 규모면 충분. 수천 명 되면 그때 집계 필드).

### 3-4. 챌린지
```
challenges/{id}  { crewId, title, kind: 'distance'|'count'|'streak', goal, from, to, uid, name, createdAt }
```
- **참여 진행도는 저장하지 않는다** — 이미 있는 `runs`에서 계산하면 된다(`lib/stats` 재사용).
  별도 progress 문서를 두면 러닝 저장과 어긋나 틀어진다. **계산으로 풀 것.**
- 당근 스크린샷의 "하루 한 번 생존 신고" 같은 인증형 챌린지는 `count` + `comments`로 표현 가능.

### 3-5. 아기자기함 (별도 트랙 · 작고 효과 큼)
회장 지적("이런 아기자기함이 없는데"). **문제는 색이 아니라 보상 루프의 부재다.**
- 당근 = 걸음 진행바 + **[받기]** + 보물상자 + 랭킹 + 마스코트
- 우리 마이 탭 = 총거리 0.0km · 러닝 0회 · **배지 전부 회색 "미획득"** → **첫인상이 "너는 아직 아무것도 아니야"**
- 처방: ①배지를 회색 무덤 대신 **"다음 배지까지 1.2km" 진행바** ②획득 순간 **축하 애니**(reanimated 이미 있음·무의존성)
  ③**마스코트 캐릭터**(로고 러너 캐릭터화, gpt-image-1) — 체감의 8할
- **당근의 광고·포인트 경제는 모방하지 않는다**(그건 당근이 돈 버는 방식이지 우리 것이 아니다).

## 4) 순서 (의존 관계가 강제하는 것)

```
P0 신원 기반 (uid + linkWithCredential + 소유 rules)   ← 전부의 전제. 안 하면 채팅=사칭·난장판
        │
        ├─ P1 일정 만들기        ← 가장 싸고, 지금 실제 병목(코드 고쳐야 일정 추가됨)
        ├─ P2 채팅              ← P1의 모임 안에서
        ├─ P3 투표              ← 채팅 위. uid 없으면 아예 성립 불가
        └─ P4 챌린지            ← runs 재사용, 저장 안 함

(병렬 가능·독립) 아기자기함 트랙 — P0 없이도 언제든
```

**지금 당장 할 것은 P0가 아니다. `DEMO=false` → main 머지가 먼저다.**
브랜치에 세 세션치가 미머지로 쌓여 있다. 여기에 커뮤니티를 얹으면 되돌리기 어려워진다.

## 5) ★ 회장 결정 3건 (이게 정해져야 코드 시작)

### 결정 A — 모임이 **하나**인가 **여럿**인가
| | 단일 크루(현재 유지) | 다중 모임(당근식) |
|---|---|---|
| 뜻 | "모두의 마라톤" 하나. 그 안에 일정·채팅·챌린지 | 사용자가 여러 모임을 만들고 가입 |
| 비용 | **작다** — 지금 IA 그대로 | **크다** — 5탭 전부 영향, 가입·권한·모임 목록·모임별 랭킹 |
| 맞는 경우 | 우리가 **하나의 러닝 크루**를 운영 | 여러 크루가 쓰는 **플랫폼**을 지향 |
| 추천 | **✅ 우선 이것** — 다중은 나중에 `crewId`만 채우면 확장 가능하게 스키마에 자리만 남겨둠 | |

> 위 스키마에 `crewId?`를 미리 넣어둔 이유가 이것이다. **지금은 안 쓰고 나중에 켠다.**

### 결정 B — 게스트도 채팅 가능?
- **가능(권장)**: 진입장벽 낮음. 단 기기 바꾸면 소유권 상실(함정 ②) → "가입하면 지켜집니다" 유도
- 불가: 채팅은 이메일 가입자만. 깔끔하지만 크루원 이탈 위험

### 결정 C — 어디부터?
- **①아기자기함 먼저**(작고 체감 큼, P0 불필요) — 지금 앱이 "초라해 보이는" 문제를 먼저 해결
- **②P0+P1(일정 만들기)** — 회장이 코드 안 고치고 일정 올릴 수 있게 되는 실질 이득
- **③P0→P2(채팅)까지 직행** — 가장 크고, 가장 늦게 효과남

## 6) 안 하기로 한 것 (명시)
- **당근의 광고·포인트 경제**: 우리 수익모델이 아님
- **챌린지 진행도 별도 저장**: `runs`에서 계산(어긋남 방지)
- **채팅 전체 구독**: 비용 폭증. `limit(50)`만
- **다중 모임 지금 착수**: 스키마에 `crewId` 자리만 남기고 보류(결정 A)
